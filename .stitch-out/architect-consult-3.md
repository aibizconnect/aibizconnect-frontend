ARCHITECTURE DECISION REQUESTED — switch Stitch/HTML import from "heuristic translation" to "lossless DOM import + Layer Tree"

Ali (after two more fidelity bugs: duplicated header band vs global header block, and giant blank bands) proposes: "can't we just copy the HTML from Stitch and put it here, but recognize the elements, sections and the whole Layer Tree? Place them one by one exactly top-to-bottom, left-to-right. Put the CSS in its place, SEO info in its place, fonts and preloading, and so on."

CONTEXT: Today's pipeline renders the page headless, annotates computed styles (data-cs), then htmlToSections HEURISTICALLY translates the DOM into our ~20 native block types (row/heading/text/button/menu/contact-form/...). We have now shipped 10+ heuristics (band expansion, outermost-grid recursion D-173, DOM-order splice D-174, column chunking D-175, button chrome D-169, sibling labels D-170, wrapper bg climb D-171...) and each user review still finds new divergences. The long tail (your D-172/RULING-172 list: gradient text, absolute decorations, pseudo-elements, srcset, blend modes, CSS vars) guarantees translation never reaches 100%.

PROPOSED MODEL (my recommendation — rule on it):
1. LOSSLESS CORE: import the page as the ACTUAL HTML document. Split top-level <section>/<header>/<footer> into our section list (so the editor's section toolbar still works — one "imported-html" section per band), each carrying its exact subtree HTML verbatim. Keep the page's real CSS: harvested <style>/<link> stylesheets + the @font-face/:root/keyframes we already extract — served scoped to the page document (preview/public routes are separate documents already; the dashboard editor renders the canvas in an iframe or with scoped CSS).
2. LAYER TREE: parse each band's subtree into an addressable tree (tag, classes, node-path, text, img src, href) shown in the editor's left rail — exactly the elements, in DOM order. Selecting a node highlights it in canvas.
3. EDITING: phase 1 node-level edits stored as a PATCH LIST keyed by stable node-path (text content, image src/alt, link href/label, inline style overrides: color/bg/font-size/padding/hide). Patches applied at render. This keeps the original HTML immutable (re-import-safe, diffable) and undo trivial.
4. SEO/FONTS: <title>/meta description/og:* → our page SEO fields; fonts → preload links + font-display:swap; images → ingest to Media Library and rewrite srcs (already built).
5. NATIVE BLOCKS COEXIST: tenant can insert our native sections between imported bands (Quick Add), and an optional per-band "Convert to editable blocks" runs the EXISTING heuristic translator when they want full block editing — the translator becomes opt-in instead of the default fidelity path.
6. Mobile fidelity comes FREE from the original responsive CSS instead of our re-derived stacking rules.

TRADE-OFFS I see: (a) shallower editability inside imported bands until the patch system matures (no drag-drop of arbitrary nodes initially — reorder limited to band level); (b) CSS isolation must be airtight in the dashboard canvas (iframe strongly preferred); (c) Tailwind runtime: Stitch ships a CDN script — we must SNAPSHOT compiled CSS at import time (our render bridge can dump document.styleSheets) so pages don't depend on a third-party CDN; (d) two storage formats (imported-html sections + native sections) — the typed-AST migration (Phase 1, in progress) should model imported-html as a first-class node type.
